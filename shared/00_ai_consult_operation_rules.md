# 00 AI相談運用ルール

> File: `shared/00_ai_consult_operation_rules.md`
> Updated: 2026-07-11
> Status: 現行運用中。

## 1. この文書の役割

本書は、AI相談における参照確認、合意、変更、レビュー、Git確定の共通運用を定義する。

技術仕様や個別環境情報は本書へ重複記載しない。

| 種類 | 正本 |
|---|---|
| 公開入口・導入 | `README.md` |
| 共通運用 | 本書 |
| 現行技術仕様 | `docs/01_current_spec.md` |
| セキュリティ | `shared/SECURITY.md` |
| ローカル環境情報 | `local/consult.local.md` |
| プロジェクト固有仕様 | 各プロジェクトの正本仕様書 |
| 現行動作の検証根拠 | 実装、試験、必要に応じて実DB |

要求、現行仕様、実装・実DB、進捗・保留、履歴は別の根拠として扱う。不一致を見つけた場合は、推測でどれかへ合わせず、不一致と影響を示して確認する。

---

## 2. 基本原則

- 現在確認できる事実だけを扱い、未確認事項を補完しない。
- 同じ仕様を複数文書へ重複記載しない。
- 現行仕様へ過去の経緯や保留事項を混在させない。
- 既存文書で扱える内容について類似文書を新設しない。
- 用途、利用者、更新条件を説明できない機能や文書を追加しない。
- 既存成果物、互換性、対象外の作業ツリーを勝手に変更しない。
- 一般論や注意事項を、確定要件として追加しない。
- 個人名、秘密情報、ローカル固有情報を公開文書へ混在させない。

---

## 3. 相談開始時の参照確認

### 3.1 start bundle

新しい相談やスレッドでは、対象プロジェクト、運用ルール、必要な仕様・実装を含む`start` bundleを参照する。

最初に以下を確認する。

- `INDEX.md`
- 対象文書・対象実装
- `PATH_INDEX.md`
- `SKIPPED.md`
- `MANIFEST.csv`
- 引き継ぎ文がある場合は目的と再開点

bundleは生成時点の参照スナップショットであり、恒久的な唯一の正ではない。新しいbundleが提示された場合は、同じ対象について古いbundleより新しい内容を優先する。

参照不足、除外、読込失敗が作業に影響する場合は、変更へ進まず必要なファイルを追加取得する。

### 3.2 パス確認

パスが不明な場合は推測しない。

1. 現行構造インデックスを`find`で検索する
2. 構造が古い場合は`structure sync`を行う
3. 対象パスを確定してから新しい`start` bundleを生成する

コマンド、設定、bundle形式の詳細は`docs/01_current_spec.md`を参照する。

### 3.3 開始時の回答

長い定型宣言は不要とする。最初の回答では、使用する参照bundle、確認できた対象、不足またはBlocking Questionだけを簡潔に示す。

## 4. Blocking Question

Blocking Questionは、回答がなければ安全または仕様上正しく変更できない場合だけ提示する。

各質問には以下を含める。

```text
BQ番号
質問
必要な理由
未確定のまま進めるリスク
```

以下はBlocking Questionにする。

- 複数の正本が矛盾し、優先判断が必要
- 破壊的変更、互換性変更、削除範囲が未確定
- 必要な実ファイル、実DB、ローカル手順が確認できない
- ユーザー判断が必要な仕様選択肢が複数ある

軽微で作業結果に影響しない不明点は、対象外として明示し、質問を増やさない。

## 5. 変更レベル

変更前にL1からL3のいずれかへ分類する。不明な場合は高いレベルを選ぶ。

| レベル | 定義 | 例 | 必要な合意 |
|---|---|---|---|
| L1 | 既存の挙動・契約・データを変えない非破壊的変更 | 誤字、リンク、説明整理、既存挙動を固定する試験追加 | 合意済みフェーズ範囲内で実施可 |
| L2 | 既存の仕様、挙動、出力、運用手順を変更するが、互換性を維持し復元可能 | 現行仕様変更、機能追加、出力変更、構造正本の再生成 | 変更内容と受入条件への明示的合意 |
| L3 | 削除、rename、移行、互換性破壊、不可逆または広範囲な変更 | 旧文書削除、互換ラッパー化、破壊的DB変更、clean export | 対象、影響、移行・復元方法への明示的合意 |

L2・L3では、変更前に最低限以下を提示する。

```text
目的
変更対象
変更しない範囲
受入条件
試験方法
```

L3では、移行順序または復元方法も提示する。

## 6. 標準フロー

```text
参照確認
→ 現状と不一致の整理
→ Blocking Question
→ 変更レベル・変更単位・受入条件の提示
→ 合意
→ 仕様文書
→ 実装
→ 試験
→ review bundle
→ commit
→ push
```

仕様または挙動を変える場合は、原則として合意済み仕様文書を先に更新してから実装する。

文書整理では、先に各文書の実在、役割、正本性、重複、不一致を確認する。確認前に削除、統合、新規作成へ進まない。

影響範囲は依頼内容に応じて確認する。無関係なAPI、DB、UI、多言語、SEOなどを定型的に列挙しない。

## 7. 文書運用

### 7.1 新規文書の作成条件

新規文書は、以下をすべて満たす場合だけ作成する。

- 既存文書とは異なる恒久的な役割がある
- 読む人と利用場面が明確
- 更新する条件が明確
- 既存文書へ記載すると役割が混在する
- 正本または案内文書から所在を確認できる

作成前に、次を提示する。

```text
文書名
役割
正本性
既存文書へ入れられない理由
更新条件
```

Phase専用の進捗文書、相談ごとのTODO、モデル別の重複文書は原則として作らない。

### 7.2 記載先

- 現在有効な仕様：現行仕様書
- 設計理由や過去の判断：設計履歴・変更履歴
- 個別PC、パス、remote、ビルド手順：`local/consult.local.md`
- 未確定事項：保留欄
- 作業中の未完了項目：既存のTODOまたはプロジェクト状況文書

同じ内容を複数箇所へ全文転記せず、必要なら正本への参照だけを置く。

### 7.3 保留事項

保留事項は確定仕様として書かない。最低限以下を記録する。

```text
項目
目的
保留理由
再開条件
再開時の最初の作業
```

解消した保留は現行仕様へ反映し、保留欄から削除または履歴へ移す。

### 7.4 TODO

TODOは現在未完了の作業と次の行動だけを保持する。

- 会話履歴を蓄積しない
- 完了済み作業を長く残さない
- commitや正本で確認できる詳細を重複記載しない
- TODO文書が存在しない案件で、相談のためだけに新設しない

### 7.5 引き継ぎ文

引き継ぎ文は次に限定する。

```text
次スレッドの目的
再開点
確定済み結果
残課題・注意事項
必要なbundle生成コマンド
```

共通運用ルールや長い完了履歴を毎回転記しない。失敗経緯は、再開判断に必要な場合だけ末尾へ記載する。

## 8. 変更適用

### 8.1 変更案

変更前に以下を確認する。

- 対象ファイルの実在
- 実在する見出し、関数、selector、アンカー
- 変更箇所
- 変更根拠
- 対象外のファイル
- Blocking Questionの有無

一次根拠の文字列をアンカーや検索条件に使う場合は、元ファイルから正確に取得する。似た文字列を推測で作らない。

### 8.2 適用スクリプト

複数ファイル、長文文書、アンカー依存の変更は、原則としてRepoRoot直下で実行する一時Pythonスクリプトとして提示する。

スクリプトは以下を満たす。

- 対象パスを固定する
- 対象ファイルの存在と想定内容またはhashを確認する
- 想定外の内容なら変更せず停止する
- 二重適用を防止する
- 書込み失敗時に部分適用を残さない
- UTF-8 BOMなし、LFで保存する
- 変更対象を出力する
- 対象外ファイルを変更しない

一時スクリプトはcommitしない。適用後に削除し、残存を確認する。

単純な確認コマンドやGit操作のためだけにスクリプトを作らない。

### 8.3 プロジェクト固有作業

DB、ビルド、deploy、公開remoteなどの手順は、プロジェクト正本または`local/consult.local.md`を根拠にする。必要な情報がない場合は推測せずBlocking Questionとする。

セキュリティと除外規則は`shared/SECURITY.md`を参照する。

## 9. 試験とreview

変更後は、変更内容に応じて構文確認、ビルド、自動試験、手動確認を行う。

受入条件は変更内容に合わせて設定する。L2・L3では正常系、異常系、境界のうち該当する観点を含める。

レビューには共通CLIの`review`を使用する。

- 対象パスを明示して無関係な変更を混ぜない
- staged、unstaged、未追跡を区別して確認する
- `DIFF_INDEX.md`、`SKIPPED.md`、`MANIFEST.csv`を確認する
- 変更ファイルの不足、意図しない変更、実装との不一致を確認する
- 問題があればcommit前に修正する

変更もskipもない場合は成果物を作らない。skipだけ存在する場合は、成果物を確認して影響を判断する。

---

## 10. Git確定

reviewと試験が完了した後、今回の対象だけをstage、commit、pushする。

- `git add .`を使用しない
- RepoRoot全体への`git add -A`を使用しない
- 新規、削除、renameでは対象を限定した`git add -A -- <path>`を使用できる
- commit前にstaged対象を確認する
- 別案件の変更、未追跡ファイル、生成物を整理・削除・stageしない
- 一時スクリプトをcommitしない
- `folder_tree.txt`は構造同期が今回の範囲に含まれる場合だけ更新する
- remote、subtree、公開手順は`local/consult.local.md`に従う
- push後はローカルHEADと対象remoteの先端を確認する

対象外の作業ツリーが存在しても、今回のパスを限定して進める。対象外変更を理由なく巻き込まない。

---

## 11. 完了条件

作業完了時に以下を確認する。

- 合意した変更だけが反映されている
- 現行仕様、実装、試験が矛盾していない
- 保留事項が確定仕様へ混入していない
- reviewで変更漏れと副作用がない
- 必要なcommitとpushが完了している
- 次の作業がある場合は、正本または簡潔な引き継ぎ文から再開できる
